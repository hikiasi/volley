VOLLEYDZEN — бренд и платформа для волейболистов всех уровней, объединяющая

ПОЛЬЗОВАТЕЛЬСКИЕ СЦЕНАРИИ
2.1 Что такое Telegram Mini App
Telegram Mini App (TMA) — это обычный веб-сайт на Next.js, который открывается внутри
мессенджера Telegram через встроенный WebView. Пользователь нажимает на кнопку в
боте — и приложение открывается прямо в Telegram, без перехода в браузер.
Авторизация происходит автоматически через Telegram-аккаунт: никаких логинов,
паролей, SMS.
ℹ️  Техническое пояснение для заказчика
С точки зрения пользователя — это мобильное приложение внутри Telegram. С точки
зрения разработки — обычный Next.js сайт с REST API и базой данных. Разница только в
том, что Telegram передаёт нам данные пользователя автоматически при открытии.

4. АВТОРИЗАЦИЯ ЧЕРЕЗ TELEGRAM WEB APP
4.1 Принцип работы
Когда пользователь открывает Mini App, Telegram SDK автоматически заполняет
переменную window.Telegram.WebApp.initData — строку с данными пользователя,
подписанную HMAC-SHA256 ключом, производным от токена бота. Это гарантирует
подлинность данных без логина/пароля.
⚠️  Безопасность
Верификация initData ОБЯЗАТЕЛЬНО выполняется на сервере (не на клиенте). Клиентский
код всегда может быть изменён через DevTools. Сервер проверяет HMAC-подпись перед
выдачей JWT.

4.2 Схема авторизационного потока
29. Пользователь открывает Mini App → Telegram SDK устанавливает
window.Telegram.WebApp.initData
30. Frontend отправляет POST /api/auth/telegram с телом { "initData": "..." }
31. Сервер верифицирует HMAC-SHA256 подпись initData через
TELEGRAM_BOT_TOKEN
32. Сервер проверяет auth_date — не старше 1 часа (защита от replay-атак)
33. Сервер ищет пользователя по telegram_id. Если не найден — создаёт
автоматически (авторегистрация)
34. Сервер возвращает JWT access token (7 дней) и данные пользователя
35. Frontend хранит JWT в памяти (React Context / Zustand store) — не в localStorage
36. Все защищённые запросы к API отправляются с заголовком Authorization: Bearer
<jwt>
37. При истечении JWT — повторная авторизация через initData (бесшовно для
пользователя)

4.3 Реализация серверной валидации
// lib/telegram.ts
import crypto from "crypto";

export interface TelegramUser {
  id: number;
  first_name: string;
  last_name?: string;
  username?: string;
  photo_url?: string;
  language_code?: string;
}

export function validateTelegramInitData(initData: string): TelegramUser | null {
  const params = new URLSearchParams(initData);

5. РОЛЕВАЯ МОДЕЛЬ: USER / TRAINER / ADMIN
5.1 Описание ролей
Роль Кто Доступ
user Обычный игрок TMA: главная, кэмпы, курсы, профиль, уведомления.
Без /admin.
trainer Тренер / куратор TMA: всё как user + кабинет тренера (ученики, видео,
отзывы, создание курсов). Без /admin.
admin Администратор TMA: всё + полный доступ к /admin (все разделы,
включая управление пользователями, финансами,
рассылками, другими admin).

5.2 Функциональность роли Trainer (куратор)
Кабинет тренера (/trainer)
• Список учеников: все пользователи, купившие курс с tier = review_lite / review_pro /
full_coaching
• Прогресс ученика: какой день, % выполнения, последняя тренировка
• Входящие видео: список видео от учеников с фильтром «не проверено / проверено»
• Форма обратной связи: текстовый комментарий + поле для ссылки на видео-ответ
• Уведомления: получает Telegram-пуш при загрузке нового видео
• Создание/редактирование курсов: полный редактор курса (недели → дни →
упражнения)

5.3 Управление ролями
• Роль назначается только в /admin → раздел «Пользователи» → кнопка «Изменить
роль»
• Новый пользователь всегда получает роль user при авторизации
• Admin может назначить любую роль: user → trainer или user → admin
• Добавление нового admin: /admin → «Администраторы» → «Добавить admin» (по
Telegram ID или username)
⚠️  Безопасность
Изменение роли на admin/trainer доступно только пользователям с ролью admin. В коде
middleware проверяет роль из JWT на каждом защищённом роуте.

9.2 Схема платёжного процесса
42. Пользователь нажимает «ОПЛАТИТЬ» → Frontend → POST /api/payments/create
43. Backend формирует запрос к ЮKassa API (POST https://api.yookassa.ru/v3/payments)
44. ЮKassa возвращает { id, status: "pending", confirmation: { url } }
45. Backend сохраняет payment в БД (status = pending), возвращает confirmation.url
46. Frontend открывает confirmation.url через Telegram.WebApp.openLink()
47. Пользователь оплачивает на странице ЮKassa
48. ЮKassa отправляет HTTP POST на /api/payments/webhook
49. Backend: проверяет IP из белого списка ЮKassa, парсит event = "payment.succeeded"
50. Транзакционно обновляет: payment.status = succeeded, booking.status = fully_paid
51. Генерирует PDF квитанцию → загружает в Timeweb S3
52. Отправляет Telegram-уведомление пользователю с ссылкой на квитанцию

  const payload: any = {
